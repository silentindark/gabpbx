========================================
chan_sofia Realtime Peers (sippeers)
========================================

chan_sofia uses the "sippeers" realtime family, compatible with
chan_sip's realtime peer table. Both channel drivers can share
the same database table.

Configuration
=============

1. /etc/gabpbx/extconfig.conf:

   [settings]
   sippeers => sqlite3,/var/lib/gabpbx/sqlite3.db

2. /etc/gabpbx/res_config_sqlite3.conf:

   [general]
   dbfile => /var/lib/gabpbx/sqlite3.db
   config_table => ast_config
   cdr_table => ast_cdr

3. /etc/gabpbx/modules.conf (preload before other modules):

   preload => res_config_sqlite3.so

Database Schema
===============

Example Usage
=============

-- Add a realtime peer:
INSERT INTO sippeers (name, secret, context, host, type, callerid)
VALUES ('rt100', 'rt100', 'phones', 'dynamic', 'friend', '"Realtime 100" <100>');

-- View registered peers:
SELECT name, ipaddr, port, datetime(regseconds, 'unixepoch', 'localtime') AS reg_time
FROM sippeers WHERE regseconds > 0;

-- Remove a peer:
DELETE FROM sippeers WHERE name = 'rt100';

Behavior
========

- Config-file peers (sofia.conf) are checked first
- If not found, chan_sofia queries the sippeers realtime table
- Realtime peers are cached in memory after first lookup
  (needed for digest auth nonce persistence)
- On successful REGISTER, the database is updated with:
  ipaddr, port, regseconds, fullcontact
- On unregistration, these fields are cleared
- Both chan_sip and chan_sofia can use the same sippeers table

Known Limitations
=================

max_contacts enforcement is per-add: a single REGISTER with multiple new Contact
headers that exceeds the limit will commit bindings for contacts before the over-limit
one, then reject the remainder with 403. In practice all common UAs send one Contact
per REGISTER, so this is a theoretical edge case only.

INVITE auth NOT IMPLEMENTED (2026-04-27): the `secret`, `md5secret`, and `insecure`
fields in sippeers are PARSED but NOT ENFORCED at runtime for inbound INVITE.
chan_sofia today accepts ALL inbound INVITEs without SIP-level auth regardless of
secret config. REGISTER auth IS implemented. Use peer ACL (permit/deny columns)
+ frontend SIP proxy (Kamailio) + TLS for inbound INVITE protection. Queued task
"post-T56 INVITE digest auth implementation" closes this gap; see CHAN_SOFIA_AUDIT.md
operational note for full disclosure.
